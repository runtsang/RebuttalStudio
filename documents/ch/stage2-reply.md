# Stage 2 — Reply

Stage 2 是真正写 Rebuttal 的阶段。针对 Stage 1 拆分出的每一条原子问题，你先用"大纲优先"的方式写出要点笔记，再点击 **Refine** 把笔记润色成正式的回复段落。语言、语气和格式交给 LLM 处理——技术内容的准确性由你负责。

## 在整体流程中的位置

```
Stage 1: Breakdown → Stage 2: Reply → Stage 3: First Round → ...
```

Stage 2 以 Stage 1 的原子问题列表为输入，为每一条生成草稿回复。Stage 3 随后将这些草稿组装成正式提交文本。

## 开始前的准备

- [ ] 当前 Reviewer 的 Stage 1 已完成——原子问题列表已经整理干净。
- [ ] 已在 **API Settings** 中配置好 API Key。
- [ ] 你已经在脑子里想清楚了每个问题的技术回答。LLM 负责润色表达，不会帮你发明数据、实验结果或引用。

> **警告：** Refined Draft 在进入 Stage 3 之前必须逐句通读。LLM 可能会误表达某个细节或引入不精确的说法，所有事实性陈述由你承担责任。

## 界面说明

**问题表格**
Stage 2 的主界面是一个表格，每行对应 Stage 1 的一条原子问题。每行包含：

- 顶部显示**问题标题**和来源标签（如 `weakness1`、`question2`）。
- **引用原文**——Reviewer 原话的逐字摘录。
- **My Reply** 文本输入框——写你的大纲草稿。
- **Refined Draft** 面板（右侧）——点击 Refine 后显示 LLM 生成的结果。
- 每行的 **Refine** 按钮——触发该行的 LLM 润色。

**My Reply 输入框**
可编辑的草稿区域。在输入框内右键单击会弹出上下文菜单，包含两个选项：

- **Insert Table**——打开弹窗，设置行列数，点击"Build Grid"生成可填写的网格，填入数据后点击"Insert"，将一个 Markdown 表格嵌入草稿。
- **Insert Code**——打开弹窗，填写可选的语言标签和代码内容，用于嵌入算法伪码或代码片段。

**Refined Draft 面板**
点击 Refine 后，LLM 生成的回复出现在此面板中，格式固定为：

```
> **Reviewer's Comment**: [Reviewer 原话]

**Response**: [润色后的正式回复]
```

这个面板也可以直接编辑——LLM 运行后，你可以也应该在这里做人工修改。

**Auto Fit 按钮**
位于中间列。点击后自动将所有行的高度调整为匹配内容高度，适合内容量差异较大时整理视图。

## 操作步骤

1. **从表格第一条问题开始，按顺序处理。**
   每行对应 Stage 1 的一条原子问题。先读顶部的引用原文，再开始写草稿。

2. **识别问题的根本关切。**
   动笔之前先想清楚：这是要求澄清？说方法有缺失？质疑方法论？还是问适用范围？搞清楚关切类型，才能写出有针对性的回复，而不是泛泛而谈。

3. **在 My Reply 里写你的大纲。**
   写 bullet points 或简短句子，涵盖你的核心要点。不需要追求正式表达——LLM 会处理这部分。好的大纲应包含：
   - 对问题的直接回答（是/否/需要看情况，以及原因）。
   - 具体证据（实验数字、表格引用、公式编号）。
   - 对 Reviewer 可能误解之处的澄清。

4. **需要时插入支撑结构。**
   在 My Reply 框内右键菜单插入 Table 或 Code：
   - 用 **Insert Table** 做横向对比（你的方法 vs. baselines，修改前后的数字对比）。
   - 用 **Insert Code** 嵌入算法伪码或配置细节。

   嵌入的表格和代码块在 Refine 后会被保留。

5. **点击 Refine。**
   LLM 接收引用原文和你的大纲，生成润色后的回复。它会选择一个合适的礼貌性开场白，将你的要点组织成清晰的段落，并输出标准的 `> Reviewer's Comment` / `**Response**:` 格式。

6. **通读并修改 Refined Draft。**
   这是最重要的一步。逐句核查生成的文本：
   - 每个数字、引用和结论是否与你的真实结果一致。
   - 你大纲里的要点有没有被遗漏。
   - 语气是否自信、直接，既不防御性也不傲慢。
   - 直接在面板中修改需要调整的内容。

   如果某个要点被漏掉，直接手动加进去，而不是重新点 Refine——重新运行会覆盖你已经做的修改。

7. **点击 Auto Fit 整理视图。**
   填完几行后，点击 Auto Fit 统一行高，便于浏览整体表格。

8. **对每一行重复上述操作。**
   Stage 2 只有在每一行都有非空的、经过人工核查的 Refined Draft 时，才算完成。

## 技巧与注意事项

- **大纲越具体，输出越准确。** 点击 Refine 之前，至少写 2–3 条有实质内容的要点。一行大纲几乎必然产出泛泛的回复；具体的技术细节（数字、引用、实验条件）才能产出精准的输出。
- **用表格呈现数字结果。** 回复"缺少某实验"类的关切时，用 Insert Table 直接在草稿里嵌入数据。Refined Draft 会保留表格结构。
- **多条 bullet = 多段回复。** 大纲中的每条 bullet point 通常会成为 Refined Draft 中的独立段落。如果你想分开处理几个子点，每个子点单独写一条 bullet。
- **大声读一遍输出。** 这是最快发现 LLM 不自然措辞、重复句式或逻辑断层的方法。
- **Stage 2 是添加内容的最后机会。** Stage 3 是格式整合，不是写作。没有在 Stage 2 处理的关切，不会出现在最终 Rebuttal 里。

## 预期输出

完成 Stage 2 意味着每一行都有非空的、经过人工核查的 Refined Draft。每条好的输出应该：

- 以一个与关切类型匹配的礼貌性开场白开头。
- 用引用块引用 Reviewer 的原话。
- 用 1–3 个聚焦段落附上具体证据来回应关切。
- 不包含虚构的数字、未核实的引用或含糊的陈述。

**需要重做的情况：** 如果 Refined Draft 完全没有回应关切的核心，或者输出质量明显不如你的大纲，检查一下大纲是否足够清晰和具体——模糊的大纲几乎总是产出模糊的输出。
